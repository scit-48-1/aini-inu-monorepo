# AINI INU PROJECT PRD

## 0. 문서 메타
- 문서명: AINI INU Project PRD
- 버전: v1.4
- 작성일: 2026-03-13
- 기준: `aini-inu-frontend`, `aini-inu-backend` 현재 구현 상태
- 목적: 현재 제품 범위, 사용자 흐름, 핵심 정책, 부분 구현 영역과 기술 제약을 단일 문서로 정리

### 0.1 문서 해석 원칙
- 이 문서는 "이상적인 미래 범위"가 아니라 "현재 코드에 반영된 제품"을 기준으로 작성한다.
- 제품 범위와 정책의 1차 설명은 `PROJECT_PRD.md`가 담당한다.
- HTTP 계약의 단일 원본은 Swagger(OpenAPI)다.
- 데이터 모델의 최종 구현 기준은 Backend Entity와 `src/main/resources/db/ddl/*.sql`이다.
- 문서와 구현이 어긋나면 먼저 PRD를 현재 코드 기준으로 정렬하고, 이후 OpenAPI/코드를 동기화한다.

### 0.2 이번 개정의 핵심 변경
- 알림, 타임라인, 실시간 산책 세션을 현재 구현 범위에 포함했다.
- 설정 화면의 `푸시 알림`, `2FA`는 서버 기능이 아니라 로컬 UI 상태임을 명시했다.
- 타 회원 프로필의 `FEED`, `HISTORY`가 아직 미구현 상태임을 반영했다.
- 실종 반려견 AI 분석, 스토리 파생 규칙, 위치 fallback, 채팅 실시간/폴링 fallback을 구현 기준으로 재정리했다.
- 리뷰가 생성되더라도 현재 백엔드에서는 매너 점수/온도를 재계산하지 않는 갭을 명시했다.

## 1. 제품 정의
### 1.1 한 줄 설명
반려견 보호자가 동네에서 산책 메이트를 찾고, 대화하고, 기록을 남기고, 긴급 실종 상황까지 연결할 수 있게 만드는 지역 기반 산책 커뮤니티 서비스.

### 1.2 해결하려는 문제
- 동네에서 함께 산책할 상대를 찾기 어렵다.
- 연결 전 상대의 신뢰도와 맥락을 파악하기 어렵다.
- 산책 경험이 기록과 관계로 누적되지 않는다.
- 실종/목격 상황에서 지역 네트워크 기반의 빠른 대응이 어렵다.

### 1.3 현재 제품이 제공하는 핵심 가치
- 발견: 주변 산책 모집글, 핫스팟, 실시간 산책중 유저를 빠르게 본다.
- 연결: 모집 참여 후 바로 채팅방으로 이어지고, 1:1 DM도 시작할 수 있다.
- 기록: 게시글, 산책일기, 스토리, 활동 타임라인으로 경험을 남긴다.
- 긴급 대응: 실종 신고와 목격 제보를 업로드하고 AI 후보 분석 후 채팅으로 연결한다.

## 2. 주요 사용자
### 2.1 Primary Persona
- 반려견 보호자
- 핵심 니즈: 산책 메이트 탐색, 안전한 연결, 산책 기록, 실종 대응

### 2.2 Secondary Persona
- 반려견 비보유 사용자
- 백엔드에서는 `NON_PET_OWNER`를 지원한다.
- 현재 프론트 회원가입 UI는 `PET_OWNER` 가입만 노출하며, 비보유 사용자 진입 경로는 제품화되어 있지 않다.

### 2.3 Emergency Persona
- 실종 반려견 보호자
- 유기견/실종견 목격 제보자
- 핵심 니즈: 이미지 업로드, 후보 분석, 빠른 연락 연결

### 3.2 현재 제품에서 부분 구현 또는 데모 성격인 항목
- 랜딩 페이지의 주소 입력 후 스캔 결과는 실제 백엔드 호출 없이 클라이언트 데모 상태로만 동작한다.
- 설정 화면의 `푸시 알림`, `2FA` 토글은 서버 저장이나 권한 제어와 연결되지 않는다.
- 타 회원 프로필의 `FEED`, `HISTORY` 탭은 "곧 준비" 상태다.
- 채팅방의 `ProfileExplorer`는 최신 프로필 화면이 아니라 레거시 `ProfileView`를 사용한다.
- Gemini 관련 프런트 보조 서비스는 저장소에 남아 있으나 실제 사용자 기능으로 연결되어 있지 않다.

### 3.3 현재 비범위
- 결제, 광고, 관리자 콘솔
- 신고/제재/모더레이션 운영도구
- 외부 푸시 인프라(APNs/FCM) 기반 알림 발송
- 고도화된 인증 보안 기능(실사용 2FA, 소셜 로그인 등)

## 5. 핵심 사용자 여정
### 5.1 신규 가입 여정
랜딩 또는 로그인 화면 -> 이메일/비밀번호/닉네임 입력 -> 즉시 토큰 발급 -> 프로필 생성 -> 반려견 등록(선택) -> 가입 완료 -> 대시보드 진입

### 5.2 산책 메이트 연결 여정
대시보드 또는 Around Me 진입 -> 주변 모집글 탐색 또는 직접 모집글 작성 -> 참여 신청 -> 채팅방 자동 연결 -> 메시지 대화 -> 산책 완료 확인 -> 리뷰 작성

### 5.3 기록 축적 여정
피드 탐색 -> 게시글 작성/댓글/좋아요 -> 프로필에서 산책일기 작성 -> 타임라인/스토리로 재노출

### 5.4 긴급 대응 여정
Around Me `EMERGENCY` 탭 진입 -> 실종 신고 또는 목격 제보 등록 -> 이미지 업로드 -> AI 분석 -> 후보 조회 -> 승인 -> `LOST_PET` 1:1 채팅방 생성 또는 재사용

### 5.5 리텐션 여정
실시간 알림 수신 -> 채팅/피드/around-me 관련 화면 재진입 -> 미작성 리뷰 처리 또는 새 상호작용 시작

## 6. 도메인별 제품 요구사항
### 6.1 인증/세션
- 이메일 기반 회원가입, 로그인, 리프레시 토큰 갱신, 로그아웃을 지원한다.
- 프런트는 refresh token만 localStorage에 저장하고 access token은 메모리에 유지한다.
- 앱 시작 시 silent refresh 후 `getMe()`로 세션을 복원한다.
- 보호 라우트는 `/dashboard`, `/feed`, `/chat`, `/around-me`, `/settings`, `/profile`이다.
- 현재 백엔드는 비밀번호를 해시가 아닌 평문 비교로 처리한다. 이는 출시 전 보안 고도화가 필요한 영역이다.

### 6.2 회원/프로필/관계
- 회원가입 직후 프로필 생성 API로 닉네임, 이미지, 자기소개, 연동 닉네임, 전화번호, 나이, 성별, MBTI, 성향, 성향 타입을 저장한다.
- 내 프로필 조회/수정, 타 회원 프로필 조회, 회원 검색, 팔로우 상태 조회/변경을 지원한다.
- 팔로워/팔로잉 목록과 카운트를 조회할 수 있다.
- 활동 통계는 최근 126일 기준으로 타임라인 이벤트를 집계해 제공한다.
- 타임라인 공개 여부는 백엔드에서 `isTimelinePublic`으로 관리하며 기본값은 공개다.

### 6.3 반려견
- 반려견 생성/수정/삭제, 메인 반려견 변경, 견종/성향/산책스타일 마스터 조회를 지원한다.
- 이름은 1~10자, 회원당 최대 10마리까지 등록 가능하다.
- 입력 canonical은 `birthDate`이며 서버에서 나이를 계산한다.
- 첫 반려견은 자동으로 메인 반려견이 된다.
- 마지막 반려견을 삭제하면 회원 타입은 `NON_PET_OWNER`로 내려간다.
- 현재 회원가입 UI는 반려견 등록을 선택적으로 넘길 수 있으므로, 프런트 기준 가입 완료 직후 "반려견이 없는 PET_OWNER" 상태가 일시적으로 가능하다.

### 6.4 대시보드
- 다음 데이터를 한 화면에서 보여준다: 활동 통계, 추천 핫스팟, 주변 최신 모집글 3건, 최근 대화 상대, 미작성 리뷰 대상.
- 일부 섹션 실패는 전체 화면 실패가 아니라 섹션별 fallback으로 처리한다.
- 주변 스레드 조회 시 geolocation을 우선 시도하고 실패하면 서울시청 좌표 fallback을 사용한다.

### 6.5 Around Me: 탐색/모집
- `FIND`, `RECRUIT`, `EMERGENCY` 3개 서브탭을 사용한다.
- 첫 진입 시 `navigator.geolocation.getCurrentPosition()`을 1회 호출한다.
- 위치 획득 실패 시 서비스 기본 좌표는 서울시청 `lat=37.566295`, `lng=126.977945`이다.
- 위치 좌표는 소수점 6자리까지 유지한다.
- 자동 주기 재탐색은 없고 사용자의 명시적 새로고침에서만 재조회한다.
- 기본 탐색 반경은 5km이며, 날짜 범위와 반경 필터를 적용할 수 있다.
- 모집글은 지도 마커, 리스트, 핫스팟 형태로 동시에 소비된다.
- 모집글 제목은 최대 30자, 설명은 최대 500자다.
- `chatType`은 `INDIVIDUAL` 또는 `GROUP`이며 생성 시 필수다.
- `INDIVIDUAL`은 정원 2명, `GROUP`은 정원 3~10명만 허용한다.
- 모집글은 시작 시각 기준 60분이 지나면 만료로 취급한다.
- 현재 활성 모집글이 있는 작성자는 추가 모집글을 만들 수 없다.
- `NON_PET_OWNER`는 모집글 생성이 불가하다.
- 참여 신청은 즉시 `JOINED`로 처리되며 채팅방이 바로 연결된다.
- 중복 신청은 idempotent success로 처리한다.
- 정원 초과 시 즉시 실패한다.

### 6.6 실시간 산책 세션
- 백엔드는 산책 시작, heartbeat, 종료, 활성 산책중 유저 목록, 내 세션 조회 API를 제공한다.
- 활성 세션은 마지막 heartbeat 후 5분이 지나면 스케줄러가 자동 종료한다.
- 산책 시작/완료는 타임라인 이벤트로 적재된다.
- 현재 프런트는 활성 산책중 유저 목록만 사용하며, 시작/종료/heartbeat를 호출하는 사용자 UI는 없다.

### 6.7 채팅
- 채팅방 목록은 `DM`, `WALK`, `LOST_PET` origin 필터를 지원한다.
- 1:1 채팅방 생성 시 기존 direct room이 있으면 재사용한다.
- 메시지 조회는 cursor 기반이며 최신 메시지부터 내려받고 화면 표시 시 시간순으로 정렬한다.
- 메시지 전송은 optimistic UI를 사용한다.
- 메시지 읽음은 debounce 후 서버에 반영한다.
- 실시간은 STOMP WebSocket을 우선 사용하고 연결 실패 시 REST polling fallback을 사용한다.
- 실시간 이벤트는 `created`, `delivered`, `read` 상태를 다룬다.
- 채팅방 나가기, 산책 완료 확인, 리뷰 작성이 구현돼 있다.
- 산책 완료 확인은 방의 모든 활성 참여자가 확인하면 스레드를 `COMPLETED`로 전환한다.
- 리뷰는 `채팅방 x 리뷰 작성자 x 리뷰 대상자` 조합으로 1회만 작성 가능하며 점수 범위는 1~5다.
- 현재 리뷰 생성은 리뷰 테이블과 리뷰 조회에는 반영되지만, 회원의 `mannerTemperature`를 재계산하는 서버 연동은 없다.

### 6.8 커뮤니티 피드
- 게시글 목록, 상세, 작성, 수정, 삭제, 댓글 목록/작성/삭제, 좋아요 토글을 지원한다.
- 게시글은 `content` 또는 `caption` 중 하나가 필수이며, 최대 2000자다.
- 이미지는 최대 5장까지 업로드할 수 있다.
- 댓글은 최대 500자다.
- 댓글 삭제는 댓글 작성자 또는 게시글 작성자만 가능하다.
- 좋아요는 프런트에서 낙관적으로 반영하고 실패 시 롤백한다.

### 6.9 산책일기/스토리/타임라인
- 산책일기는 산책 기록의 원본 객체이며 CRUD를 지원한다.
- 산책일기 생성 조건은 다음과 같다: `COMPLETED` 상태의 스레드여야 하고, 작성자는 해당 스레드 작성자 또는 참여자여야 하며, 동일 회원은 동일 스레드에 산책일기를 1회만 작성할 수 있다.
- 산책일기 제목은 최대 120자, 본문은 최대 300자, 사진은 최대 5장이다.
- 공개 여부 기본값은 공개이며 비공개로 변경 가능하다.
- 타인의 일기 조회는 공개 일기만 가능하다.
- 스토리는 독립 도메인이 아니라 팔로잉 중인 사용자의 "최근 24시간 내 공개 산책일기"를 파생 조회한 결과다.
- 스토리 노출 규칙은 다음과 같다: 작성자를 팔로우한 사용자만 조회 가능하며, 삭제되지 않은 공개 산책일기만 포함하고, 작성자별 1개 그룹으로 묶고, 그룹 내부 정렬은 `createdAt desc`다.
- 타임라인은 산책 모집글 생성, 실시간 산책 시작/완료, 게시글 생성, 실종 신고 생성, 목격 제보 생성, 산책일기 생성 이벤트를 적재한다.
- 타인의 비공개 타임라인은 403을 반환한다.
- 현재 프런트에서 타임라인 공개 설정을 변경하는 UI는 없다.

### 6.10 실종 반려견
- 실종 신고 생성, 내 신고 목록/상세 조회, 목격 제보 생성, AI 분석, 후보 조회, 후보 승인 흐름을 지원한다.
- 실종 신고는 동일 회원 기준 `petName + breed`의 active duplicate를 차단한다.
- 목격 제보 생성 시 AI 인덱싱은 비동기 보조 처리이며 실패해도 제보 등록 자체는 유지된다.
- AI 분석은 실종 신고 owner만 수행할 수 있다.
- 분석 성공 시 검색 세션과 후보 snapshot을 저장한다.
- 기본 검색 세션 TTL은 24시간, 후보 상위 노출 기본값은 20건이다.
- AI 분석 실패 시 `L500_AI_ANALYZE_FAILED`를 반환하고 세션을 생성하지 않는다.
- 후보 승인 시 `LOST_PET` direct chat room 생성을 시도한다.
- 채팅 생성 실패 시 매칭은 `PENDING_CHAT_LINK` 상태로 남을 수 있다.

### 6.11 알림
- 알림 목록 조회, unread count 조회, 개별 읽음 처리, 전체 읽음 처리를 지원한다.
- STOMP `/ws/notifications`를 통해 사용자별 실시간 알림을 푸시한다.
- 현재 알림 발생원은 새 채팅 메시지, 산책 완료 확인, 산책 참여 신청, 게시글 댓글 4가지다.
- 프런트는 사이드바/탭바에서 unread badge와 패널 UI를 제공한다.
- 설정 화면의 푸시 토글은 현재 알림 발송 로직을 제어하지 않는다.

### 6.12 설정
- 라이트/다크 테마 토글을 제공한다.
- 로그아웃을 제공한다.
- `푸시 알림`, `2FA` 토글 UI가 있으나 현재는 `zustand persist` 기반 로컬 상태일 뿐 서버 기능이 아니다.

## 8. 기술/아키텍처 요약
### 8.1 프런트엔드
- Next.js 16 App Router
- React 19
- Tailwind CSS v4
- Zustand 기반 클라이언트 상태 관리
- STOMP WebSocket
- Leaflet 지도
- Daum 주소 검색

### 8.2 백엔드
- Spring Boot 3.5
- Java 21
- Spring Data JPA
- PostgreSQL
- WebSocket/STOMP
- JWT 기반 인증
- Spring AI + pgvector 기반 실종 반려견 분석 보조

### 8.3 저장/업로드
- 커뮤니티 이미지 업로드는 presigned token 후 로컬 파일 저장 방식이다.
- 기본 로컬 스토리지 경로는 `../common-docs/storage`다.

### 8.4 이벤트성 데이터 흐름
- 타임라인은 도메인 이벤트를 받아 저장한다.
- 알림은 DB 저장 후 STOMP로 푸시한다.
- 실시간 채팅은 WebSocket과 REST fallback을 병행한다.

## 9. 현재 구현 갭과 후속 정렬 과제
### 9.1 제품 기능 갭
- 타 회원 프로필 `FEED`, `HISTORY` 탭 미구현
- 실시간 산책 세션 시작/종료/heartbeat 사용자 UI 없음
- 타임라인 공개 설정 UI 없음
- 설정의 푸시/2FA 토글은 로컬 상태만 변경
- 랜딩의 "동네 스캔"은 데모 흐름

### 9.2 구조/기술 갭
- 채팅 프로필 탐색 패널이 레거시 `ProfileView`를 사용
- `src/services/api/*`, 구형 회원가입 단계 컴포넌트 등 레거시 코드가 일부 잔존
- 리뷰 생성 후 회원 `mannerTemperature` 갱신 로직 미연결
- 인증 비밀번호 해시 미적용

## 11. 요약
AINI INU의 현재 구현은 "동네 산책 탐색 + 모집 + 채팅 + 피드 + 프로필/일기 + 실종 반려견 AI 분석"까지는 MVP를 넘어서는 수준으로 연결돼 있다. 다만 설정 서버 연동, 타회원 프로필 일부 탭, 실시간 산책 세션 제어 UI, 리뷰-매너점수 연결, 보안 고도화는 아직 정리되지 않은 상태다. 본 문서는 앞으로의 제품 논의에서 "이미 구현된 것"과 "아직 제품화되지 않은 것"을 구분하는 기준 문서로 사용한다.
