## 2. 제품 정의

AINI-INU는 반려견 보호자 중심의 지역 기반 소셜 서비스이며, 핵심 기능은 다음 네 축으로 구성된다.

1. 계정, 프로필, 반려견 등록 기반의 사용자 식별
2. 주변 산책 메이트 모집, 산책 진행, 산책 기록 작성
3. 채팅, 후기, 커뮤니티 피드, 팔로우 기반의 관계 형성
4. 실종견 신고, 목격 제보, AI 유사도 분석 기반의 재회 지원

## 3. 사용자 유형과 권한

### 3.1 회원 유형
- `PET_OWNER`: 반려견 보호자. 현재 회원가입 UI 기본값.
- `NON_PET_OWNER`: 백엔드 enum 지원. 현재 일반 회원가입 UI에서는 직접 노출되지 않음.
- `ADMIN`: 백엔드 enum 지원. 운영/관리 목적.

### 3.2 회원 상태
- `ACTIVE`
- `INACTIVE`
- `BANNED`

### 3.3 접근 제어
- 인증 필요 경로: `/dashboard`, `/feed`, `/chat`, `/around-me`, `/settings`, `/profile`
- 비인증 사용자는 보호 경로 접근 시 `/login`으로 리다이렉트되어야 한다.
- 본인 프로필과 타인 프로필은 서로 다른 화면 구성을 사용한다.
- 타인 비공개 타임라인 접근 시 403을 반환하고 프론트는 잠금 상태 UI를 표시해야 한다.

## 5. 공통 시스템 요구사항

### 5.1 API 계약
- 프론트는 `/api/v1`를 기본 경로로 사용해야 한다.
- 서버 응답은 `ApiResponse<T>` 래퍼를 사용해야 한다.
- 프론트 API 클라이언트는 래퍼를 해제한 실제 데이터를 반환해야 한다.
- 클라이언트 타임아웃은 8초 기준으로 동작해야 한다.
- 401 발생 시 refresh token 기반 재시도 큐를 사용해야 한다.

### 5.2 인증/세션
- 인증은 JWT access token + refresh token 구조여야 한다.
- 앱 초기 진입 시 세션 복구를 시도해야 한다.
- 로그인 상태 확인 실패 시 보호 경로에서는 로그인 페이지로 이동해야 한다.
- 로그아웃 시 로컬 인증 상태를 제거하고 서버 로그아웃도 시도해야 한다.

### 5.3 오류 처리
- 네트워크 오류와 타임아웃은 재시도 액션이 포함된 토스트로 노출해야 한다.
- 비즈니스 오류는 백엔드가 내려주는 한국어 메시지를 우선 사용해야 한다.
- 주요 화면은 `loading / empty / error / success` 상태를 명시적으로 가져야 한다.
- 부분 실패가 가능한 화면은 전체 실패 대신 섹션 단위 오류와 재시도를 지원해야 한다.

### 5.4 이미지 업로드
- 업로드는 presigned URL 방식이어야 한다.
- 순서:
  1. 업로드 URL 요청
  2. 응답에서 업로드 토큰/URL 확보
  3. 바이너리 PUT 업로드
  4. 최종 public image URL을 기능 API에 전달
- 허용 파일:
  - 최대 10MB
  - `image/jpeg`
  - `image/png`
  - `image/webp`

### 5.5 위치 정보
- 위도/경도는 소수점 6자리 정밀도로 정규화해야 한다.
- 위치 권한 실패 또는 위치 미확보 시 서울시청 좌표 `37.566295, 126.977945`를 기본값으로 사용한다.

## 6. 도메인별 상세 요구사항

## 6.1 인증 및 회원가입

### 6.1.1 로그인
- 이메일과 비밀번호로 로그인해야 한다.
- 로그인 실패 시 인라인 오류를 표시해야 한다.
- 일반적인 페이지 레벨 크래시 UI로 대체하면 안 된다.

### 6.1.2 회원가입 플로우
- UI는 다음 4단계를 유지해야 한다.
  1. `ACCOUNT`
  2. `PROFILE`
  3. `PET`
  4. `COMPLETE`
- 현재 프론트는 회원가입 요청에 `memberType: PET_OWNER`를 전송한다.
- 반려견 등록 단계에는 항상 "나중에 등록하기" 선택지가 있어야 한다.

### 6.1.3 회원가입 입력 제약
- 이메일: 필수, 이메일 형식
- 비밀번호: 8~64자, 영문 대문자/소문자/숫자/특수문자 포함
- 닉네임: 2~10자, 한글/영문/숫자만 허용

### 6.1.4 인증 관련 엔드포인트 요구사항
- 로그인, 로그아웃, refresh, 현재 사용자 조회를 지원해야 한다.
- 테스트 인증용 `/api/v1/test/auth/token`은 외부에서 호출 가능하지만 운영 노출 정책을 문서로 통제해야 한다.
- `/api/v1/test/auth/me`는 숨김 API로 취급한다.

## 6.2 회원 프로필 및 관계

### 6.2.1 프로필 기본 정보
- 사용자는 본인 프로필을 조회/수정할 수 있어야 한다.
- 타인 프로필은 공개 정보만 조회 가능해야 한다.
- 프로필 필드 제약:
  - `linkedNickname`: 최대 30자
  - `phone`: `01x-xxxx-xxxx` 계열 형식 허용
  - `age`: 1~100
  - `mbti`: 최대 4자
  - `personality`: 최대 50자
  - `selfIntroduction`: 최대 200자

### 6.2.2 프로필 화면 구성
- 본인 프로필은 다음 섹션을 포함해야 한다.
  - 프로필 수정
  - 반려견 목록
  - 팔로워/팔로잉 수
  - 게시글
  - 산책 일지/히스토리
  - 후기
  - 타임라인
  - 활동 히트맵
- 타인 프로필은 현재 다음 상태를 가진다.
  - `TIMELINE`, `REVIEWS`: 구현됨
  - `FEED`, `HISTORY`: 플레이스홀더 상태

### 6.2.3 팔로우 관계
- 사용자는 다른 회원을 팔로우/언팔로우할 수 있어야 한다.
- 현재 팔로우 여부를 조회하는 전용 엔드포인트가 필요하다.
- 프론트는 팔로우 수를 낙관적 업데이트해야 한다.
- 팔로워/팔로잉 목록은 공개 조회 가능해야 한다.

### 6.2.4 후기
- 회원 후기 목록 조회가 가능해야 한다.
- 후기 작성은 채팅방 단위/대상 회원 단위로 중복 방지되어야 한다.
- 평점은 1~5만 허용한다.

### 6.2.5 활동 통계
- 활동 통계는 본인 및 특정 회원 기준으로 조회 가능해야 한다.
- 현재 런타임 계약은 `stats/activity`를 사용한다.
- 기존 OpenAPI의 `stats/walk` 정의는 현행 코드와 불일치한다.

## 6.3 반려견 관리

### 6.3.1 반려견 등록/수정/조회
- 사용자는 반려견을 등록, 조회, 수정, 삭제할 수 있어야 한다.
- 산책 모집 작성 시 자신의 반려견 중 하나 이상을 선택해야 한다.
- 반려견이 없는 사용자는 산책 모집 작성이 차단되어야 한다.

### 6.3.2 반려견 입력 제약
- `name`: 1~10자
- `breedId`: 필수
- `birthDate`: 필수
- `gender`: 필수
- `size`: 필수
- `mbti`: 최대 4자
- `isNeutered`: 필수
- 나이는 직접 입력이 아니라 `birthDate` 기반 파생값으로 취급해야 한다.

## 6.4 대시보드

### 6.4.1 목적
- 대시보드는 사용자의 오늘 활동과 주요 액션을 한 화면에서 보여주는 홈이어야 한다.

### 6.4.2 구성 데이터
- 활동 통계
- 주변 인기 산책 지점 또는 hotspot
- 근처 모집 중 산책
- 내 반려견 목록
- 최근 대화 상대
- 산책 기반 후기 대기 목록

### 6.4.3 동작 요구사항
- 위치 기반 데이터는 현재 위치 또는 서울시청 fallback으로 조회해야 한다.
- 각 섹션은 독립적으로 실패/재시도 가능해야 한다.
- 근처 산책 카드 클릭 시 `/around-me?threadId={id}`로 deep-link 되어야 한다.
- 후기 대기 항목은 강제 팝업이 아니라 모달 진입 방식이어야 한다.
- 최근 친구 목록은 상대 회원 기준으로 중복 제거되어야 한다.
- 상단 히어로는 사용자 닉네임과 대표 반려견 정보를 사용하되, 데이터 누락 시 fallback 문구를 가져야 한다.

## 6.5 주변 산책 찾기 및 모집

### 6.5.1 화면 구조
- `FIND`
- `RECRUIT`
- `EMERGENCY`

### 6.5.2 위치 취득
- 페이지 진입 시 1회 `navigator.geolocation.getCurrentPosition`으로 위치를 시도해야 한다.
- 자동 주기 갱신은 하지 않는다.
- 새로고침은 수동 액션으로만 이뤄져야 한다.

### 6.5.3 산책 모집 조회
- 필터:
  - `dateFrom`
  - `dateTo`
  - `radius`
- 사이드바 페이지네이션 이동 시 위치/날짜/반경 필터가 유지되어야 한다.
- 대시보드 deep-link로 전달된 `threadId`가 있으면 해당 상세를 자동 선택해야 한다.

### 6.5.4 산책 모집 생성/수정
- 모집 생성과 수정은 동일 폼을 재사용할 수 있어야 한다.
- 입력 제약:
  - `title`: 필수, 1~30자
  - `description`: 필수, 1~500자
  - `walkDate`: 필수
  - `startTime`: 필수
  - `chatType`: 필수
  - `maxParticipants`: 2~10
  - `petIds`: 1개 이상 필수
  - 위치: `placeName`, `latitude`, `longitude` 필수
- `chatType`은 `INDIVIDUAL` 또는 `GROUP`이며 프론트 기본값은 `INDIVIDUAL`이다.
- 프론트는 제목, 날짜, 시간, 반려견 선택, 설명 길이 제한을 선검증해야 한다.
- 주소 검색은 Daum Postcode를 사용하고, 좌표 변환은 위치 서비스 레이어를 통해 수행해야 한다.

### 6.5.5 활성 산책 스레드 제약
- 사용자는 동시에 활성 산책 모집을 중복 보유할 수 없어야 한다.
- 단, 기존 모집 수정 흐름에서는 활성 스레드 가드가 `editingThreadId` 존재 시 우회 가능해야 한다.
- 내 활성 모집과 내가 참여한 모집을 별도 조회할 수 있어야 한다.

### 6.5.6 산책 스레드 상태
- `RECRUITING`
- `EXPIRED`
- `COMPLETED`
- `DELETED`

## 6.6 산책 세션

### 6.6.1 목적
- 실제 산책 시작, 진행, 종료를 트래킹하는 별도 런타임 기능이 필요하다.

### 6.6.2 필수 동작
- 산책 시작 API가 필요하다.
- 이미 활성 세션이 있으면 새 시작은 충돌 처리되어야 한다.
- 클라이언트는 1~2분 간격 heartbeat를 보내야 한다.
- 서버는 heartbeat가 5분 이상 없으면 세션을 자동 종료해야 한다.
- 활성 산책 중인 사용자 목록 조회 API가 필요하다.
- 내 세션 목록 조회 API가 필요하다.
- 산책 시작/종료는 타임라인 이벤트로 기록되어야 한다.

### 6.6.3 프론트 요구사항
- 현재 산책 중 사용자 드롭다운은 `/walking-sessions/active` 데이터를 사용해야 한다.
- 목록은 자동 갱신이 아니라 수동 새로고침 기반이어야 한다.
- 경과 시간 표시는 1분 주기로 갱신되어야 한다.

## 6.7 산책 일지 및 스토리

### 6.7.1 산책 일지
- 산책 일지는 산책 스레드를 기반으로 작성해야 한다.
- 작성 가능 스레드 목록 조회 API가 필요하다.
- 입력 제약:
  - `threadId`: 필수
  - `title`: 1~120자
  - `content`: 1~300자
  - `photoUrls`: 최대 5개
  - `walkDate`: 필수
  - `isPublic`: 미지정 시 `true`

### 6.7.2 산책 기록 활용
- 본인 프로필에서 산책 히스토리로 노출되어야 한다.
- 타임라인 이벤트로 연결될 수 있어야 한다.

### 6.7.3 스토리
- 스토리는 작성자 기준으로 그룹핑되어야 한다.
- 작성자별 스토리 수량 배지가 표시되어야 한다.
- 스토리 영역에서 모달 진입이 가능해야 한다.

## 6.8 채팅 및 산책 후기

### 6.8.1 채팅방 구분
- 탭:
  - `DM`
  - `WALK`
  - `LOST_PET`
- 목록은 서버에서 `origin`과 `status=ACTIVE`로 필터링되어야 한다.
- 검색은 클라이언트에서 방 이름, 제목, 마지막 메시지를 대상으로 수행한다.

### 6.8.2 채팅방 상태
- `ACTIVE`
- `CLOSED`

### 6.8.3 메시지 처리
- 메시지 내용은 1~500자여야 한다.
- 클라이언트는 `messageType: USER`로 전송해야 한다.
- 메시지 전송은 낙관적 pending bubble을 사용해야 한다.
- 실패한 메시지는 재시도 버튼을 제공해야 한다.
- 읽음 처리는 방 진입 후 디바운스 2초 기준으로 수행해야 한다.
- 읽음 상태는 상대방이 실제 읽은 경우에만 `READ`로 표현해야 한다.
- 기본 표시는 "전송완료"여야 한다.

### 6.8.4 메시지 조회
- 방 전환 시 이전 방 메시지는 초기화해야 한다.
- 최초 진입 시 다음 데이터를 함께 불러와야 한다.
  - 채팅방 정보
  - 최근 메시지
  - 산책 확정 상태
  - 내 후기 작성 여부
- 이전 메시지 로딩은 cursor 기반 역방향 infinite scroll이어야 한다.

### 6.8.5 실시간성
- WebSocket STOMP를 기본으로 사용해야 한다.
- 폴링 fallback은 5초 주기로 유지되어야 한다.
- 창 visibility 변화와 알림 이벤트는 목록 갱신 트리거가 되어야 한다.

### 6.8.6 빠른 입력 UX
- 입력창은 500자 제한을 표시해야 한다.
- 빠른 답변 chip을 제공할 수 있어야 한다.

### 6.8.7 방 액션
- 채팅방 나가기는 확인 모달을 거쳐야 한다.
- `WALK` 채팅방에서는 산책 확정/취소가 가능해야 한다.
- 산책 확정 후 후기 작성 모달 진입이 가능해야 한다.

### 6.8.8 후기 정책
- 후기 점수는 1~5만 허용한다.
- 동일 방/대상 조합에서 중복 후기 작성은 막아야 한다.

## 6.9 커뮤니티 피드

### 6.9.1 피드 목록
- 피드는 `loading / error / empty / success` 상태를 가져야 한다.
- 무한 스크롤을 지원해야 한다.

### 6.9.2 게시글 작성/수정
- 백엔드 제약:
  - `content` 또는 `caption` 중 하나는 필수
  - 최대 2000자
  - `imageUrls`: 최대 5개
- 현재 프론트 제약:
  - 이미지 최소 1개 필수
  - 내용 비어 있으면 생성 불가
- 수정 시에도 비어 있는 내용은 허용되지 않아야 한다.

### 6.9.3 게시글 인터랙션
- 좋아요는 낙관적 업데이트 후 실패 시 롤백해야 한다.
- 댓글 펼치기 시 게시글 상세를 다시 조회해야 한다.
- 댓글 삭제 권한:
  - 댓글 작성자
  - 게시글 작성자
- 게시글 수정/삭제 권한:
  - 게시글 작성자

### 6.9.4 댓글 제약
- 댓글 길이: 1~500자

## 6.10 실종견 신고 및 AI 매칭

### 6.10.1 긴급 제보 화면
- 모드는 `LOST`, `FOUND` 두 가지여야 한다.
- 두 모드 모두 이미지 업로드가 필수다.
- `LOST` 모드:
  - 반려견 이름 필수
  - 위치 필수
- `FOUND` 모드:
  - 위치 필수

### 6.10.2 신고 생성 플로우
- `LOST`:
  1. 이미지 업로드
  2. 실종 신고 생성
  3. 분석 후속 플로우 진입
- `FOUND`:
  1. 이미지 업로드
  2. 목격 제보 생성

### 6.10.3 AI 분석
- 분석 요청은 `lostPetId`가 필수다.
- `image` 또는 `imageUrl` 중 하나는 필수다.
- 좌표는 6자리 정밀도로 정규화되어야 한다.
- PRD/플로우 기준 점수 가중치:
  - 유사도 70
  - 거리 20
  - 최신성 10

### 6.10.4 매칭 정책
- 실종 신고 생성 후 AI 분석이 자동으로 트리거되어야 한다.
- AI 분석 실패 시 `L500_AI_ANALYZE_FAILED` 오류를 사용자에게 노출해야 한다.
- 실패 상태에서 수동 재시도 버튼은 제공하지 않는다.
- 후보 정렬은 검색 세션 기준 순서를 유지해야 한다.
- 사용자가 매칭 승인해야만 이후 채팅방 생성으로 진행할 수 있다.

### 6.10.5 실종 신고 상태
- `ACTIVE`
- `RESOLVED`
- `CLOSED`

## 6.11 알림 및 타임라인

### 6.11.1 알림
- 프론트는 벨 아이콘과 알림 패널을 제공해야 한다.
- 알림 목록 조회, 읽음 처리, 전체 읽음 처리가 가능해야 한다.
- 읽음 처리는 낙관적 업데이트가 가능해야 한다.
- 실시간 알림은 `/ws/notifications`와 `/user/queue/notifications`를 사용해야 한다.
- 미읽음 수는 전역 스토어에 보관해야 한다.

### 6.11.2 알림 타입
- `CHAT_NEW_MESSAGE`
- `WALK_CONFIRM`
- `WALK_APPLICATION`
- `COMMENT_ON_POST`

### 6.11.3 알림 라우팅
- 알림 유형에 따라 다음 화면으로 이동해야 한다.
  - 채팅
  - 주변 산책
  - 피드
  - 대시보드

### 6.11.4 타임라인
- 회원별 타임라인 조회가 가능해야 한다.
- 본인 타임라인 공개 설정 변경 API가 필요하다.
- 타임라인은 날짜별 그룹핑으로 렌더링해야 한다.
- 403 응답은 비공개 타임라인으로 해석해야 한다.
- 확인된 이벤트 유형:
  - 산책 모집 생성
  - 산책 세션 시작
  - 산책 세션 종료
  - 게시글 작성
  - 실종견 신고
  - 목격 제보
  - 산책 일지 작성

## 6.12 설정

### 6.12.1 현재 구현 상태
- 테마 전환은 `next-themes` 기반이어야 한다.
- push 알림 토글은 현재 로컬 상태만 관리한다.
- 2FA 토글은 현재 로컬 상태만 관리한다.
- 설정 스토어는 `aini-inu-config` 키로 localStorage에 저장된다.

### 6.12.2 주의사항
- push, 2FA는 서버 영속화 요구사항으로 아직 확정되지 않았고 현재 런타임 계약도 없다.
- 따라서 설정 화면은 부분적으로 UI 선구현 상태로 간주한다.

## 7. 비기능 요구사항

### 7.1 성능 및 UX
- 주요 목록 화면은 skeleton 또는 명시적 로딩 상태를 제공해야 한다.
- 실시간 기능은 WebSocket을 우선 사용하고 fallback 경로를 가져야 한다.
- 낙관적 UI는 롤백 가능해야 한다.
- 위치 기반 기능은 권한 거부 시에도 기본값으로 동작을 이어가야 한다.

### 7.2 보안
- 보호 API는 JWT 인증이 필요해야 한다.
- 업로드는 presigned URL을 통해 직접 스토리지 전송해야 한다.
- Nginx는 업로드 용량 제한과 API rate limit을 적용해야 한다.

### 7.3 운영
- 루트 `docker-compose.yml` 기준으로 `postgres`, `backend`, `frontend`, `nginx`가 함께 구동 가능해야 한다.
- Nginx 라우팅 요구사항:
  - `/api/v1/*` -> backend
  - `/ws/*` -> backend
  - `/swagger-ui/` -> backend
  - `/v3/api-docs` -> backend
  - `/` -> frontend
- WebSocket 업그레이드 헤더를 지원해야 한다.

### 7.4 기술 스택 제약
- 백엔드: Spring Boot 3.5.x, Java 21
- AI/검색: Gemini embeddings, PgVector
- 프론트: Next.js App Router 기반 React

## 9. 계약 및 문서 불일치 사항

### 9.1 공유 OpenAPI 스냅샷 누락
- 다음 런타임 엔드포인트가 `common-docs/openapi/openapi.v1.json`에 반영되지 않았거나 불완전하다.
  - `/api/v1/members/me/reviews`
  - `/api/v1/members/{memberId}/reviews`
  - `/api/v1/members/me/stats/activity`
  - `/api/v1/members/{memberId}/stats/activity`
  - `/api/v1/threads/my/active`
  - `/api/v1/threads/my/joined`
  - `/api/v1/walk-diaries/available-threads`
  - `/api/v1/walking-sessions/start`
  - `/api/v1/walking-sessions/heartbeat`
  - `/api/v1/walking-sessions/stop`
  - `/api/v1/walking-sessions/active`
  - `/api/v1/walking-sessions/my`
  - `/api/v1/notifications`
  - `/api/v1/notifications/unread-count`
  - `/api/v1/notifications/{id}/read`
  - `/api/v1/notifications/read-all`
  - `/api/v1/members/{memberId}/timeline`
  - `/api/v1/members/me/timeline/settings`

### 9.2 경로 명세 불일치
- OpenAPI 스냅샷은 `stats/walk`를 포함하지만 실제 구현은 `stats/activity`를 사용한다.
- 팔로우 상태 조회 GET 메서드는 런타임에 존재하지만 스냅샷 반영이 불완전하다.

### 9.3 프론트/백엔드 정책 불일치
- 회원가입 UI는 `PET_OWNER`만 선택하도록 사실상 고정되어 있다.
- 게시글 생성은 백엔드상 텍스트만으로 가능하지만 프론트는 이미지 1장 이상을 강제한다.
- 설정의 push/2FA는 UI는 있으나 서버 저장 계약이 없다.
- 타인 프로필의 `FEED`, `HISTORY`는 화면만 존재하고 실제 기능은 미완성이다.

### 9.4 운영 문서 불일치
- `GCP_DEPLOYMENT_GUIDE.md`는 GCP VM + Docker Compose 배포를 설명한다.
- `.github/workflows/deploy.yml`은 `main` 브랜치 push 시 EC2에 백엔드 JAR를 배포한다.
- 즉, 현재 저장소의 문서상 운영 방식은 단일 경로로 정합화되어 있지 않다.

## 10. 요구사항 확정이 필요한 항목

### 10.1 제품 정책 측면
- `NON_PET_OWNER` 회원가입을 실제 서비스에서 허용할지
- 게시글 작성 시 이미지 필수 정책을 유지할지
- push/2FA를 서버 영속화 기능으로 승격할지
- 타인 프로필 `FEED`, `HISTORY`를 실제 범위에 포함할지

### 10.2 계약 정비 측면
- 공유 OpenAPI를 현행 백엔드 기준으로 재생성할지
- 활동 통계 경로를 `stats/activity`로 통일할지
- 알림/타임라인/산책 세션을 공식 PRD 범위로 승격할지

## 11. 결론

현재 저장소는 공식 PRD 범위를 대부분 구현하고 있으며, 그 위에 산책 세션, 알림, 타임라인 같은 런타임 확장 기능이 추가된 상태다. 따라서 앞으로의 요구사항 기준 문서는 "공식 범위"와 "실제 구현 범위"를 분리 관리해야 하며, 특히 공유 OpenAPI 스냅샷과 운영 문서의 최신화가 우선 과제다.
