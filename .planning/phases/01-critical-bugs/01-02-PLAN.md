  <action>
    Apply minimal defensive patches to each file. Read each file fully before modifying. Changes should be surgical -- do not refactor logic, just add guards.

    **A. useRadarLogic.ts -- Consecutive failure counting to stop infinite polling**
    - Add a `useRef` for `failCountRef` initialized to 0
    - In the catch block of fetchData: increment `failCountRef.current`
    - If `failCountRef.current >= 3`: clear the interval (stop polling), log `console.warn('[useRadarLogic] Polling stopped after 3 consecutive failures')`
    - On successful fetch: reset `failCountRef.current = 0`
    - Do NOT change the polling interval, effect dependencies, or any other logic

    **B. chat/[id]/page.tsx -- Message polling failure backoff**
    - Add a `useRef` for `failCountRef` initialized to 0
    - In the catch block of the message polling function: increment `failCountRef.current`
    - If `failCountRef.current >= 3`: clear the polling interval, log `console.warn('[ChatRoom] Message polling stopped after 3 consecutive failures')`
    - On successful message fetch: reset `failCountRef.current = 0`
    - Add optional chaining to ALL nested property accesses from API data: `currentRoom?.partner?.id`, `currentRoom?.partner?.nickname`, etc.
    - Guard `rooms.find(...)` -- if rooms is falsy, default to `undefined` (e.g., `const currentRoom = rooms?.find(...)`)
    - Do NOT change WebSocket logic, message sending, or any other feature code

    **C. layout.tsx -- Auth guard hardening**
    - The existing try-catch is already good. Add one additional guard: after `JSON.parse(raw)`, check that `db` is actually an object (`typeof db === 'object' && db !== null`) before accessing `db.currentUserId`. If not an object, treat as unauthenticated and redirect to login.
    - This is a one-line addition inside the existing try block.

    **D. dashboard/page.tsx -- Replace blank returns with fallback UI**
    - Find `if (!userProfile) return null` and replace `null` with a minimal centered loading/fallback div: `<div className="flex items-center justify-center h-full"><p className="text-zinc-400">Loading...</p></div>`
    - Add optional chaining to any `Object.values(...)` calls on API data: wrap in `try-catch` or guard with `Array.isArray()` / type check before using `.filter()`
    - Add `?? []` or `?? {}` defaults where data from `Promise.allSettled` results is destructured

    **E. around-me/page.tsx -- Replace blank return with loading state**
    - Find `if (!mounted || isLoading) return null` and replace `null` with a minimal loading indicator: `<div className="flex items-center justify-center h-full"><p className="text-zinc-400">Loading...</p></div>`

    **F. feed/page.tsx -- Guard Object.entries on API data**
    - Find any `Object.entries(followingDiaries)` or `Object.values(...)` calls on API response data
    - Add guards: if the value is not an object/array as expected, default to empty `{}` or `[]`
    - Add optional chaining on nested property access chains from API data

    **G. profile/[memberId]/page.tsx -- Defensive null guards**
    - Add optional chaining (`?.`) on all nested property accesses from API-sourced data
    - Add default values for destructured properties from hooks (e.g., `const { data = null, error = null } = ...`)
    - Wrap any direct API call in the component with try-catch if not already wrapped

    **IMPORTANT SCOPE RULE (per CONTEXT.md locked decision):** Do NOT fix any API URL, HTTP method, or payload shape. Only add defensive wrapping (try-catch, optional chaining, null guards, default values, failure counting). If you notice a mismatch, leave the code as-is -- it will be cataloged in Task 2.
