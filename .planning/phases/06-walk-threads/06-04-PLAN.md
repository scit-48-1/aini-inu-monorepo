**Fix 1 — WalkThreadPetRepository.java:**

Replace the existing `deleteAllByThreadId` method with a JPQL-based version annotated with `@Modifying(clearAutomatically = true)`. Add required imports. The full interface becomes:

```java
package scit.ainiinu.walk.repository;

import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.data.jpa.repository.Modifying;
import org.springframework.data.jpa.repository.Query;
import org.springframework.data.repository.query.Param;
import scit.ainiinu.walk.entity.WalkThreadPet;

import java.util.List;

public interface WalkThreadPetRepository extends JpaRepository<WalkThreadPet, Long> {

    List<WalkThreadPet> findAllByThreadId(Long threadId);

    @Modifying(clearAutomatically = true)
    @Query("DELETE FROM WalkThreadPet wtp WHERE wtp.threadId = :threadId")
    void deleteAllByThreadId(@Param("threadId") Long threadId);
}
```

Root cause: derived `void delete*` methods do NOT get the Hibernate 1st-level cache cleared automatically. After the bulk DELETE the entity manager still has the old WalkThreadPet objects in memory. When `saveThreadPets` calls `save()` on new WalkThreadPet objects with the same (threadId, petId), Hibernate tries to merge the cached stale entities, hits the DB unique constraint, and throws. `clearAutomatically = true` forces a flush + evict of the affected entities after the DELETE.
