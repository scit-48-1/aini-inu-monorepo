<objective>
Fix follow state not persisting after refresh (Gap 2) and NeighborsModal showing wrong member's list (Gap 3).

Purpose:
- Gap 2: `fetchFollowState` in OtherProfileView scans `getFollowing({size:100})` to determine follow state. If the logged-in user follows more than 100 people, the target may not appear and `isFollowing` initializes to `false`. The backend has a dedicated endpoint `GET /members/me/follows/{targetId}` that returns the exact follow status. This must be used instead.
- Gap 3: `getFollowers()`/`getFollowing()` always call `/members/me/followers` and `/members/me/following`. When OtherProfileView opens NeighborsModal, it shows the logged-in user's list, not the profile owner's. The backend supports `/members/{memberId}/followers` and `/members/{memberId}/following`. Adding optional `memberId` routing and passing `memberId` from OtherProfileView fixes the display. Additionally, `followerCount` in OtherProfileView is hardcoded to 0; it should come from the member's actual data from `getMember()`.

Output: Updated members.ts, OtherProfileView.tsx, and NeighborsModal.tsx.
</objective>
